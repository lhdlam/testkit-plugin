Tài liệu · Hướng dẫn cho tester

testkit — AI tự viết bộ test automation

Triết lý: agent viết & bảo trì test, Playwright/pytest chạy test. Bạn (tester) là người gác cổng: ở mỗi pha bạn review artifact rồi mới cho agent đi tiếp. Quy trình đi qua 6 pha, mỗi pha kết thúc bằng một artifact + một điểm review của con người.

Dùng sidebar bên trái để chuyển trường hợp sử dụng, hoặc click từng bước (step) để nhảy nhanh. Mỗi bước có Tại sao · Bạn làm · Bạn thấy để người mới follow được.

Trước khi bắt đầu & cài đặt plugin

testkit chạy trên Claude Code (gate cưỡng chế) và Cursor (gate advisory). Agent viết test; framework chạy test trỏ tới URL môi trường deploy (Staging/UAT) — testkit không dựng app ở local.

Phiên bản hiện tại hiện thực đầy đủ 2 target: web-playwright (web app, test trỏ URL Staging/UAT) và desktop-pyside6 (app PySide6, test in-process bằng pytest-qt). Hướng dẫn chính đi theo ví dụ web; xem UC2 cho desktop.

0. Cần chuẩn bị

  • Claude Code (hoặc Cursor) cài plugin testkit.
  • Node.js 18+ (node --version).
  • URL môi trường deploy: STAGING_URL (và UAT_URL nếu có). testkit KHÔNG dựng app ở local.
  • Tài khoản test riêng cho automation trên Staging (tách khỏi tài khoản người thật).
  • (Khuyến nghị) Playwright MCP để agent mở Staging xác thực selector. KHÔNG cần để chạy test — chỉ để agent tìm/verify selector. Bắt buộc cho web-from-docs/web-blackbox, khuyến nghị cho web-playwright, không dùng cho desktop. Cài nhanh: /testkit:mcp.

Mô hình tinh thần: 6 pha, mỗi pha 1 cổng review

6 pha giống nhau cho mọi nền tảng; khác biệt gói trong profiles/<target>.md. Mỗi pha tạo ra một artifact và có một điểm review của con người trước khi sang pha sau.

Pipeline Từ init đến CI — 🚦 = bạn review trước khi sang pha sau

testkit · 6-phase pipeline
/testkit:init Phase 0 Setup: scaffold + CLAUDE.md + config (không cần review) │ /testkit:analyze Phase 1 Hiểu app → feature-map.md 🚦 bạn review │ /testkit:cases Phase 2 Sinh test case + rtm.md 🚦 bạn review │ /testkit:scenarios Phase 3 Map selector thật + kịch bản E2E 🚦 bạn review │ /testkit:script Phase 4 Sinh Page Object + *.spec.ts + auth 🚦 bạn đọc assertion │ /testkit:run Phase 5 Chạy + phân loại fail + self-heal (lặp tới khi xanh) │ /testkit:ci Phase 6 Sinh workflow CI (xong)

Cơ chế cổng PENDING → APPROVED — bạn đổi tay dòng cuối artifact để mở pha sau

e2e-tests/docs/*.md · review gate
# Mỗi artifact trong e2e-tests/docs/ kết thúc bằng một dòng: > Review: PENDING # Sau khi bạn đọc và đồng ý, ĐỔI TAY thành: > Review: APPROVED # Trên Claude Code, nếu pha trước còn PENDING thì hook pre-tool-gate # sẽ CHẶN pha sau và nhắc bạn. Đây chính là chỗ bạn kiểm soát chất # lượng — đừng duyệt cho có. Trên Cursor gate là advisory (tự giác).

Pipeline đầy đủ (web-playwright) — ví dụ module Login

Đi trọn 6 pha cho một web app: agent đọc code → hiểu app → sinh test case → map selector thật → sinh test code Playwright (TypeScript, Page Object) → chạy & tự chữa → sinh CI. Test trỏ URL Staging/UAT; auth đăng nhập 1 lần rồi tái dùng phiên.

Bắt đầu nhỏ: làm trọn module login (analyze → … → run xanh) trước khi nhân rộng. Vừa kiểm chứng quy trình, vừa cho agent một "mẫu" để bắt chước cho các module sau.

Xem test chạy thật — /testkit:watch

Khi muốn nhìn test chạy để demo hoặc soát case (cần máy có màn hình). CI thì vẫn luôn headless — watch chỉ để debug/demo.

Watch UI mode · headed · slow-mo

watch · chỉ để debug/demo
▼ web — Playwright UI mode (tốt nhất để soát coverage) $ cd e2e-tests && TEST_ENV=staging npx playwright test --ui ▼ web — mở trình duyệt thật, chậm, 1 luồng $ TEST_ENV=staging SLOWMO=500 npx playwright test --headed --workers=1 --grep @smoke ▼ desktop — hiển thị app + slow-mo $ UI_SLOWMO=500 pytest -s tests/test_login.py

Luật vàng · lệnh nhanh · sự cố thường gặp

Luật vàng Đừng phá

2 luật vàng
1. KHÔNG green-washing. Test đỏ vì app sai → đó là test làm đúng việc. Đừng để agent "sửa test cho xanh". Quy ước này nằm sẵn trong CLAUDE.md; thấy agent định làm yếu assertion → DỪNG. 2. Dữ liệu tự sinh, tự dọn. Staging dùng chung → mỗi test tạo dữ liệu riêng (email có timestamp), không đụng dữ liệu người khác. Flaky do data drift chữa bằng cách này, KHÔNG bằng tăng timeout.

Lệnh nhanh Chạy suite

cd e2e-tests
$ TEST_ENV=staging npx playwright test # full $ TEST_ENV=staging npx playwright test --grep @smoke # chỉ smoke (critical path) $ TEST_ENV=uat npx playwright test # đổi môi trường bằng 1 biến $ npx playwright show-report # xem báo cáo HTML $ npx playwright show-trace # mổ xẻ 1 test fail

Subagents (anti-cheat & chất lượng)

  • test-integrity — quét diff test tìm green-washing (assertion bị làm yếu/xoá, skip, timeout phình).
  • selector-stability — bắt selector brittle (XPath/CSS/nth) hoặc thiếu objectName.
  • coverage-auditor — dựng/kiểm RTM, báo yêu cầu chưa có test.
  • failure-classifier — phân loại mỗi fail (Bug | Selector | Timing | Data | Env | Doc-drift), route bug → bugs.md.
